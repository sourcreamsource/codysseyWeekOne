## 🟢 container writable layer / volume / bind mount 비교표  
- container writable layer = 컨테이너 실행 시 image 위에 추가되는 **쓰기 레이어(writable layer)**  
- volume = Docker가 관리하는 **별도 영속 저장소**  
- bind mount = 호스트의 실제 폴더/파일을 컨테이너에 직접 연결하는 방식  

| 항목          | container writable layer      | volume                       | bind mount               |  
| ----------- | ----------------------------- | ---------------------------- | ------------------------ |
| 정체          | 컨테이너가 실행되며 생기는 **쓰기 가능 레이어**  | Docker가 생성/관리하는 **별도 저장공간**  | 호스트의 **실제 경로**를 컨테이너에 연결 |  
| 데이터 위치      | 컨테이너의 writable layer          | Docker가 관리하는 volume 영역 내부    | 사용자가 지정한 호스트 경로          |  
| 누가 관리       | Docker가 컨테이너와 함께 관리           | Docker가 직접 관리                | 사용자가 직접 관리               |  
| 컨테이너 삭제 시   | 보통 함께 사라짐                     | **남길 수 있음**                  | 원본은 호스트에 있으므로 안 사라짐      |  
| 영속성         | 약함                            | 강함                           | 강함                       |  
| 호스트에서 직접 보기 | 매우 불편함                        | 직접 볼 수는 있지만 보통 Docker 기준으로 관리 | 매우 쉬움                    |  
| 다른 컨테이너와 공유 | 부적합                           | 적합                           | 가능하지만 경로 의존성 큼           |  
| 이식성         | 낮음                            | 상대적으로 높음                     | 낮음                       |  
| 주 사용처       | 임시 실행 결과, 캐시, 컨테이너 내부 기본 변경사항 | DB 데이터, 업로드 파일, 장기 보관 데이터    | 소스코드 공유, 설정파일 주입, 개발용 작업 |  
| 성능/권장성      | 영속 저장용으로 비권장                  | Docker가 **권장하는 기본 영속 저장 방식** | 개발 편의성은 좋지만 호스트 의존성 큼    |  
| 위험 요소       | 컨테이너 삭제하면 데이터 손실 가능           | Docker가 관리하므로 상대적으로 안전       | 컨테이너가 호스트 파일을 직접 수정 가능   |  



<br><br>

## 🟢 한 줄 정리  
- **container writable layer** 는 컨테이너 안에서만 잠깐 쓰는 저장공간에 가깝다.  
- **volume** 은 Docker가 관리하는 영속 저장공간이라 운영용 데이터 저장에 적합하다.  
- **bind mount** 는 호스트 폴더를 바로 연결하므로 개발 중 수정 반영 확인에 가장 편하다.
