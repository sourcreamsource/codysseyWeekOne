## 🟢 미션의 핵심 목적 분석  
- 이번 1주차 미션은 단순히 Docker 명령 몇 개를 실행하는 과제가 아니라, 개발 워크스테이션을 스스로 구축하고 그 과정을 문서화하는 훈련이다.  
- 핵심 도구는 터미널, Docker, Git/GitHub 이며, 각 도구를 따로 배우는 것이 아니라 하나의 개발 흐름 안에서 연결해서 다루는 것이 중요하다.  
- 요구사항 문서를 보면 실습 자체보다도 실행 결과를 증거로 남기고, 왜 이런 구조가 필요한지 설명 가능한 상태가 되는 것을 더 중요하게 본다.  
- 즉, 이번 과제의 목표는 "명령어를 따라 친 사람"이 아니라 "환경을 만들고 검증하고 설명할 수 있는 사람"이 되는 것이다.  


<br>

### 🟡 이 미션에서 실제로 해야 하는 일  
```bash
1. 터미널 기본 조작 익히기
2. 파일/디렉토리 권한 확인 및 변경
3. Docker 설치와 동작 확인
4. 이미지와 컨테이너 관리 실습
5. Dockerfile로 커스텀 이미지 제작
6. 포트 매핑 확인
7. bind mount / volume 검증
8. Git 설정 및 GitHub 연동
9. 전체 과정을 README에 정리
```

- 따라서 결과물은 코드 몇 개보다도, 학습 과정 전체가 정리된 기술 문서의 완성도가 중요하다.  
- README는 단순 소개 문서가 아니라 수행 보고서 역할도 해야 한다.  



<br><br>

## 🟢 최종 제출물 분석  
- 제출의 중심은 GitHub Repository 링크 하나이다.  
- 하지만 실제 평가 대상은 저장소 안에 포함된 기술 문서, 실습 로그, Dockerfile, 접속 증거, 데이터 영속성 검증 기록까지 전부 포함된다.  
- 즉 저장소 링크만 열어도 평가자가 전체 수행 내용을 따라가며 확인할 수 있어야 한다.  


<br>

### 🟡 최종 결과물에 반드시 포함되어야 하는 요소  
```bash
1. README 또는 기술 문서
2. 터미널 명령어와 출력 기록
3. Docker 설치 및 점검 로그
4. Dockerfile 기반 웹 서버 컨테이너
5. 포트 매핑 접속 증거
6. 바인드 마운트 반영 증거
7. Docker volume 영속성 증거
8. Git 설정 및 GitHub/VSCode 연동 증거
9. 트러블슈팅 2건 이상
```

- 이 요구사항을 보면 단순 성공 화면만 있으면 안 되고, "어떤 명령으로 무엇을 확인했는지"가 드러나야 한다.  
- 특히 Docker volume과 bind mount는 둘 다 저장과 관련되지만 목적이 다르므로 비교 설명이 필요하다.  
- 또한 README만 봐도 재현 가능해야 하므로, 경로, 포트, 사용한 명령, 결과 해석까지 적는 것이 중요하다.  



<br><br>

## 🟢 기능 요구사항 세부 분석  


<br>

### 🟡 요구사항을 작업 단위로 나누면 다음과 같다  
```bash
A. CLI 기본기
- pwd, ls -la, cd, mkdir, cp, mv, rm
- cat 또는 파일 내용 확인
- touch 로 빈 파일 생성

B. 권한 실습
- 파일 1개 권한 변경
- 디렉토리 1개 권한 변경
- 변경 전/후 비교 기록

C. Docker 기본 실습
- docker --version
- docker info
- docker images
- docker ps / docker ps -a
- docker logs
- docker stats

D. 컨테이너 실행 실습
- hello-world 실행
- ubuntu 컨테이너 진입
- ls, echo 같은 간단 명령 실행
- attach / exec 차이 관찰

E. Dockerfile 실습
- 베이스 이미지 선택
- 커스텀 포인트 설명
- 빌드 및 실행 성공

F. 네트워크/스토리지 실습
- 포트 매핑
- bind mount
- Docker volume

G. Git/GitHub 실습
- git config 설정
- git config --list 기록
- GitHub 연동 증거
```

- 즉 요구사항은 크게 CLI, 권한, Docker 운영, Dockerfile 제작, 네트워크 연결, 스토리지 영속성, Git 협업 준비로 구성된다.  
- 이 중에서 가장 중요한 검증 포인트는 "실제로 동작했는지"이다.  
- 예를 들어 포트 매핑은 단순히 `docker run -p` 명령만 적는 것이 아니라, 브라우저 접속 결과까지 있어야 인정된다.  
- volume도 단순 생성으로 끝나는 것이 아니라, 컨테이너 삭제 전후 데이터가 유지되는지를 반드시 보여줘야 한다.  


<br>

### 🟡 요구사항에서 특히 중요한 해석 포인트  
```bash
image != container
container 삭제 != volume 데이터 삭제
port mapping = host와 container 연결
Git = 로컬 버전 관리
GitHub = 원격 저장소 및 협업 플랫폼
```

- 평가자는 단순 실행 여부뿐 아니라 위 개념 차이를 이해했는지도 함께 볼 가능성이 높다.  
- 따라서 문서에는 명령어 나열만 하지 말고, 왜 이 명령을 썼는지 한두 줄씩 정리하는 것이 필요하다.  



<br><br>

## 🟢 제약 사항 및 주의점 분석  
- 모든 작업은 터미널 기반으로 수행해야 한다.  
- Dockerfile은 직접 작성해야 하므로 자동 생성 도구나 단순 복붙만으로 끝내면 학습 효과가 떨어진다.  
- 브라우저 접속 증거에는 주소창과 포트가 보여야 하고, 로그 캡처에는 명령어와 출력이 같이 있어야 한다.  
- 가장 중요한 보안 제약은 토큰, 비밀번호, 개인키, 인증코드 같은 민감정보를 문서와 이미지에 남기지 않는 것이다.  


<br>

### 🟡 환경 제약까지 고려한 해석  
```bash
서울캠퍼스 환경에서는 sudo 사용이 제한될 수 있음
따라서 Docker 실행 환경으로 OrbStack 사용 가능
터미널에서는 일반 docker 명령 그대로 사용
```

- 이 부분은 설치 방식보다 Docker 엔진이 정상 동작하는가가 더 중요하다는 뜻이다.  
- 즉 Docker Desktop, OrbStack, Colima 중 무엇을 쓰더라도 요구사항에서 원하는 검증 명령은 동일하게 수행할 수 있어야 한다.  
- 또한 README는 특정 개인 환경에만 종속되면 안 되므로, 내 환경에서만 되는 경로나 설정이 있다면 대체 방법도 같이 적어야 한다.

### 🟡 과제 완료 후 반드시 말로 설명할 수 있어야 하는 내용  
```bash
1. 절대 경로와 상대 경로 차이
2. 파일 권한 rwx 의미
3. 755 / 644 해석 방법
4. Dockerfile 기반 커스텀 이미지 제작 과정
5. 포트 매핑이 필요한 이유
6. Docker volume 이 필요한 이유
7. Git 과 GitHub 의 역할 차이
```

- 이 목록은 사실상 평가 기준이기도 하다.  
- 따라서 각 실습 문서를 작성할 때는 결과만 남기는 것이 아니라, 나중에 설명 가능한 문장으로 정리해야 한다.  
- 특히 이번 주차의 핵심 개념은 "재현 가능한 실행 환경", "격리된 실행", "데이터 영속성", "협업 가능한 버전 관리"로 요약할 수 있다.  
- 결국 이번 과제는 개발 도구를 설치하는 과제가 아니라, 개발 환경을 구축하고 검증하고 기록하는 사고방식을 익히는 과제라고 볼 수 있다.
