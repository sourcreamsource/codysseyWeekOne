# 🟩 왜 컨테이너 내부 포트로 직접 접속 할 수 없으며, 포트매핑이 필요한 이유는?  

<br>

## 🟢 1.컨테이너 내부 포트로 직접 접솔 할 수 없는 이유  
- Linux가 만들어준 별도의 네트워크 공간(네임스페이스) 안에 있기 때문에 그 IP/포트 자체를 모름.  
- 외부에서 격리된 네트워크 스페이스를 찾지 못하는 것임.  

<br><br>

## 🟢 2. 포트매핑을 하는 이유  
- 외부에서 접속할 때에는 호스트까지만 바로 온다. 하지만,  
- 호스트 포트를 Docker 포트에 연결해줘야 외부에서 호스트포트로 접속 시 Docker 포트까지 연결되어 접속이 가능하다  
- 호스트와 컨테이너 사이에 번역기(NAT) 를 놓는 것이다.  
- 호스트 포트를 컨테이너 포트로 연결하는 NAT 규칙을 만들어야 패킷이 컨테이너까지 도달할 수 있다.

# 🟩 현재 나의 OS에서 사용중인 Port 확인 방법  

> docker에서 포트 지정을 했는데.. “호스트 포트가 이미 사용 중”이라 포트 매핑이 실패한다면, 어떤 순서로 원인을 진단할지 설명할 수 있는가?  

<br>

## 🟢 포트의 역할과 과정  

### 🟡 포트는 뭐냐  

포트는 **컴퓨터 안에서 어떤 프로그램에게 트래픽을 넘길지 구분하는 번호**다.  
예를 들어 같은 컴퓨터 안에 웹서버, DB, 앱서버가 동시에 떠 있어도 포트 번호로 구분한다.  

예:  
* 80 → 웹서버  
* 443 → HTTPS  
* 3306 → MySQL  
* 5432 → PostgreSQL  

<br>

### 🟡 서버와 클라이언트  
* **서버**: 기다리는 쪽  
* **클라이언트**: 접속하는 쪽  

예:  
* nginx → 서버  
브라우저가 `localhost:6463`으로 접속하면  
서버가 그 포트에서 기다리고 있어야 연결된다.  

<br>

### 🟡 🔥 LISTEN은 정확히 무슨 상태냐  

TCP 서버는 대충 이 순서로 동작한다.  

* 소켓 생성  
* 포트에 묶음 (`bind`)  
* **기다림 (`listen`)**  
* 들어온 연결 받음 (`accept`)**  

리눅스 매뉴얼도 이 순서를 그대로 설명한다.  
`bind()`로 로컬 주소/포트를 정하고, `listen()`으로 들어오는 연결을 받을 준비를 하며, `accept()`로 실제 연결을 받는다.  



<br><br><br><br>

## 🟢 그림으로 보기  

### 🟡 아직 연결 전  

```text
브라우저(클라이언트)  ----접속 시도---->  서버 프로그램(nginx)  
                                         ^  
                                         |
                                   6463 포트에서 LISTEN 중  
```

뜻:  

* nginx가 6463 포트를 열어두고  
* 누가 오면 받을 준비를 하고 있음  
* 아직 실제 대화는 시작 전  

<br>

### 🟡 연결 후  

```text
브라우저  <------ 데이터 주고받는 중 ------>  nginx  
```

이때는 더 이상 “기다리기만 하는 상태”가 아니라  
실제 연결이 성립된 상태다.

## 🟢 포트 확인하는 명령어  

### 🟡 가장 쉬운 포트 확인 명령어  
lsof -i :특정포트번호  

```bash
lsof -i :6463  
```

### 🟡 mac  
```bash
# 사용중인 포트 확인하기  
lsof -nP -iTCP -sTCP:LISTEN  

# 특정 포트 확인하기  
lsof -nP -iTCP:6463 -sTCP:LISTEN  
```

### 🟡 linux  
```bash
# 사용중인 포트 확인하기  
ss -ltnp  

# 특정 포트 확인하기  
ss -ltnp 'sport = :6463'  
```
